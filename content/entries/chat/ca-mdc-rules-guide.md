### ❓ Что это

Правила `.mdc` — контекстные микро-инструкции, которые триггерятся динамически только для
конкретных файлов и модулей проекта по паттерну (`globs`), а не применяются глобально ко всей
кодовой базе сразу. В отличие от общего `CLAUDE.md`, который читается целиком при старте любой
задачи, `.mdc`-правило подгружается только когда агент реально работает с файлом, попадающим под
указанный паттерн — например, только при правке файлов фронтенд-компонентов.

### 🎯 Зачем тебе

Проект с разнородными частями, где единые правила для всего репозитория избыточны или даже неверны
для отдельных модулей — например, «всегда используй CSS-переменные из tokens.css» релевантно только
при правке компонентов, но бессмысленно (и тратит контекст впустую) при работе с backend-кодом на
Python. `.mdc`-правила держат контекст релевантным конкретной части работы.

### 💻 Минимальный пример

```markdown
---
description: Применяется при создании/правке фронтенд-компонентов
globs: src/components/**/*, **/*.tsx, **/*.css
---

Используй только CSS-переменные из tokens.css для цветов, не хардкодь HEX-значения напрямую.
```

Это правило подключается автоматически только когда агент трогает файлы, подпадающие под указанные
glob-паттерны — не расходует контекст на не связанные с фронтендом задачи.

### ⚠️ Грабли

- **Слишком широкий glob-паттерн** делает правило почти таким же глобальным, как обычный CLAUDE.md,
  теряя смысл точечного триггера — держи паттерн узким и специфичным.
- **Пересекающиеся правила из разных `.mdc`-файлов** на один и тот же путь могут давать
  противоречивые инструкции — избегай дублирования областей действия между разными правилами.
- **Не заменяет линтер/форматтер** — правило это инструкция для модели, а не автоматическая проверка
  на уровне инструмента, для гарантированного применения стиля нужны ещё и hooks (см. Hooks в Claude
  Code).
