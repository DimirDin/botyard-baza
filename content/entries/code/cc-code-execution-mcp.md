### ❓ Что это

Комбинация code execution tool и MCP: вместо того чтобы грузить в контекст полные определения сотен
тулов от десятков подключённых MCP-серверов, агент зовёт MCP-серверы **кодом**, исполняемым в
песочнице — обращения идут программно, а не через отдельный tool_use-ход на каждый вызов, что
экономит и токены на описания тулов, и число round-trip'ов диалога.

### 🎯 Зачем тебе

Проект с большим числом подключённых MCP-серверов (Git, Postgres, Slack, внутренние корпоративные
интеграции) — без этого механизма контекст раздувается описаниями всех доступных тулов ещё до
первого реального вопроса пользователя. Code execution с MCP переворачивает это: агент видит
компактный интерфейс и вызывает нужное программно, как обычные функции внутри скрипта.

Помимо экономии токенов на описаниях, программный вызов даёт то, чего отдельные tool_use-ходы не
дают напрямую — обычные конструкции языка программирования: циклы, условия, промежуточные переменные.
Задача «пройтись по последним 50 коммитам и для тех, что содержат fix, показать diff» без code
execution означала бы либо 50 отдельных ходов диалога, либо надежду, что модель правильно
сориентируется в длинном списке вручную; со скриптом это обычный цикл, исполняемый за один шаг.

### 💻 Минимальный пример

Концептуально, вместо цепочки отдельных tool_use-ходов «вызвать git_log → получить результат →
вызвать git_diff → получить результат» агент пишет один скрипт:

```python
commits = git_log(limit=5)
for c in commits:
    if "fix" in c.message:
        print(git_diff(c.sha))
```

исполняемый целиком в песочнице, где `git_log`/`git_diff` — прокси к реальным MCP-инструментам.

### ⚠️ Грабли

- **Требует, чтобы MCP-сервер поддерживал такой режим вызова** — не любая интеграция автоматически
  совместима, зависит от конкретной реализации моста между code execution и MCP-протоколом.
- **Отладка сложнее, чем явная цепочка отдельных tool_use** — ошибка внутри скрипта менее прозрачна
  по шагам, чем последовательность отдельных, явно залогированных вызовов.
- **Не для тулов с необратимыми side-эффектами без паузы** — как и с programmatic tool calling
  в целом, цикл, вызывающий изменяющие состояние MCP-тулы много раз подряд, требует явных
  ограничений.
