### ❓ Что это

Claude Code умеет самостоятельно выполнять полный git-цикл: создавать ветки под задачу, писать
осмысленные коммит-сообщения по конвенции проекта (по умолчанию следует стилю недавних коммитов в
репозитории, если явно не задан другой формат), и оформлять pull request'ы с заполненным описанием —
опираясь на реальный `git diff`/`git log`, а не на придуманное описание того, что «должно было»
измениться.

### 🎯 Зачем тебе

Экономия рутинного ручного шага после каждой сессии — не нужно самому формулировать коммит-сообщение,
разбираться, какие файлы стейджить, писать описание PR с чек-листом тестирования. Особенно ощутимо на
частых небольших итерациях, где ручное оформление коммита занимает сравнимое с самой правкой время.

### 💻 Минимальный пример

```
> закоммить эти изменения и открой PR
```

Агент выполняет `git status`/`git diff` для понимания реального объёма изменений, формирует коммит-
сообщение, отражающее суть (не просто «update files»), и вызывает `gh pr create` с телом,
структурированным по Summary/Test plan.

### ⚠️ Грабли

- **Никогда не должен коммитить и пушить без явного запроса** — это деструктивное по значимости
  действие, у Claude Code есть встроенная осторожность здесь, но при кастомных хуках/permissions
  стоит явно проверить, что автокоммит не включён «по умолчанию всегда».
- **Force push и `--no-verify` — явно запрещённые по умолчанию операции** — если агент предлагает их
  как способ обойти упавший pre-commit хук, это сигнал остановиться и разобраться в причине, а не
  разрешать обход.
- **Коммит-сообщение по стилю проекта требует, чтобы в истории вообще был последовательный стиль** —
  на репозитории с хаотичной историей коммитов агенту не на что ориентироваться, стоит явно задать
  формат в CLAUDE.md.
