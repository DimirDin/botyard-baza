### ❓ Что это

Подключение MCP-сервера, работающего внутри Docker, к Claude Code — даёт агенту возможность читать
логи контейнеров, перезапускать сервисы, управлять локальными Docker Compose стеками через
инструменты, а не через ручные `docker` команды в Bash tool напрямую. Отдельный MCP-слой поверх
Docker API даёт более структурированный и безопасный интерфейс, чем прямой shell-доступ к сокету
Docker.

### 🎯 Зачем тебе

Локальная разработка с многосервисным Docker Compose стеком (как этот проект — postgres, redis,
backend, bot) — вместо того чтобы просить агента вручную собирать `docker compose logs -f backend`
и разбирать вывод, MCP-сервер даёт структурированный доступ: список контейнеров, их статус, логи по
фильтру, перезапуск конкретного сервиса — без риска, что агент случайно тронет что-то за пределами
явно разрешённого Docker-контекста.

### 💻 Минимальный пример

```json
{
  "mcpServers": {
    "docker": { "command": "docker-mcp-server", "args": ["--compose-file", "docker-compose.yml"] }
  }
}
```

После подключения агент может спросить «почему backend не отвечает» и сам вызвать инструмент
получения логов конкретного сервиса вместо общей команды `docker ps` в Bash.

### ⚠️ Грабли

- **Доступ к Docker-сокету — по сути root-доступ к хосту** — MCP-сервер, управляющий Docker, должен
  быть настроен с тем же уровнем осторожности, что и прямой доступ к системе, ограничивай scope
  явно, если это возможно в конкретной реализации сервера.
- **Не для продовых Docker-хостов без дополнительных мер** — то, что удобно локально для дебага
  compose-стека разработки, не должно напрямую применяться к продовой инфраструктуре без отдельного
  слоя авторизации и аудита.
- **Пересечение с обычным Bash tool** — если у агента и так есть Bash доступ к `docker` командам,
  MCP-сервер добавляет структуру, но не единственный способ достичь того же результата — не
  дублируй логику проверки прав в двух местах по-разному.
