### ❓ Что это

Model Context Protocol — клиент-серверная архитектура на JSON-RPC 2.0, разделяющая логику модели
(что и как рассуждает Claude) от логики внешних инструментов (как получить данные из Postgres,
прочитать файл, дёрнуть Slack API). MCP-клиент (Claude Code, Claude Desktop, любое другое приложение
с поддержкой протокола) подключается к одному или нескольким MCP-серверам, каждый из которых
объявляет доступные ему **tools** (действия с side-эффектами), **resources** (данные для чтения) и
опционально **prompts** (готовые шаблоны запросов).

#### Почему протокол, а не просто API

До MCP каждая интеграция ИИ-приложения с внешним источником данных писалась заново для каждой пары
«приложение × источник» — N приложений и M источников давали N×M интеграций. MCP стандартизует
интерфейс: один сервер, написанный один раз под конкретный источник (например, Postgres), работает с
любым MCP-совместимым клиентом без переписывания под каждый.

### 🎯 Зачем тебе

Понимание архитектуры полезно, если пишешь собственный MCP-сервер (не просто подключаешь готовые) —
знание разделения tools/resources/prompts и транспортного уровня (stdio для локальных серверов,
HTTP/SSE для удалённых) определяет, как правильно спроектировать интерфейс своей интеграции, а не
изобретать велосипед поверх протокола.

### 💻 Минимальный пример

Упрощённая схема сообщения JSON-RPC при вызове инструмента:

```json
{ "jsonrpc": "2.0", "id": 1, "method": "tools/call",
  "params": { "name": "query_db", "arguments": { "sql": "SELECT ..." } } }
```

Сервер отвечает результатом или ошибкой в стандартном JSON-RPC формате — клиент (Claude) не знает
внутреннего устройства сервера, только протокольный контракт.

### ⚠️ Грабли

- **Локальные (stdio) серверы исполняются как процессы на машине клиента** — те же риски, что у
  Bash tool: сервер с полным доступом к файловой системе даёт агенту столько же возможностей,
  сколько получил бы сам процесс сервера.
- **Протокол не диктует безопасность, только транспорт** — авторизация, ограничение прав, аудит
  вызовов — ответственность конкретной реализации сервера, не встроенная гарантия MCP как такового.
- **Версионирование протокола** — MCP развивается, и старые клиенты/серверы могут быть несовместимы
  по фичам (например, tool search как отдельная возможность появился не в первой версии спеки).
