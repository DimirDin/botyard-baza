### ❓ Что это

Hooks — конфигурируемые shell-команды, которые Claude Code запускает автоматически на определённые
события жизненного цикла сессии: перед вызовом инструмента, после вызова, при старте/остановке
сессии. В отличие от инструкции в промпте («не забудь прогнать линтер после правок»), которую модель
может просто забыть выполнить, хук — гарантированное срабатывание на уровне харнеса, не зависящее от
того, «вспомнила» ли модель.

### 🎯 Зачем тебе

Проверки и форматирование, которые должны выполняться **всегда**, а не только когда агент решит, что
это уместно: автоформатирование файла сразу после каждой правки (`PostToolUse` хук на Edit), запрет
на коммит без прохождения тестов (`PreToolUse` хук на git commit), логирование каждого Bash-вызова
для аудита. Это системная гарантия, а не просьба к модели — модель физически не может «забыть»
исполнить хук, потому что она его не исполняет, это делает харнесс.

### 💻 Минимальный пример

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit",
      "hooks": [{ "type": "command", "command": "prettier --write $CLAUDE_FILE_PATH" }]
    }]
  }
}
```

После каждой правки файла инструментом Edit автоматически запускается `prettier` — вне зависимости
от того, попросил ли пользователь отформатировать код в этом конкретном запросе.

### ⚠️ Грабли

- **Хуки исполняются с правами твоего пользователя**, как любой shell-скрипт — плохо написанный хук
  из скопированного откуда-то конфига может быть таким же риском, как непроверенный MCP-сервер.
  Проверяй feedback хуков как исходящий от пользователя, но сам код хука — как обычный shell-код.
- **Хук, который падает с ошибкой, может заблокировать весь цикл** — если `PreToolUse` хук завершается
  с ненулевым кодом, действие может быть отменено целиком, тестируй хуки отдельно перед включением.
- **Не путай с permissions** — permissions решают, разрешено ли действие вообще (с подтверждением
  или без), hooks выполняют побочное действие вокруг уже разрешённого события, это разные механизмы.
