### ❓ Что это

Bash tool даёт Claude прямой доступ к shell: запуск команд, чтение вывода, работа с файловой
системой. Модель не «представляет», что вывела бы команда — она реально её исполняет в
предоставленном окружении (контейнер/VM, который настраиваешь ты) и видит настоящий stdout/stderr.

Это тот же примитив, на котором построен Claude Code изнутри — Bash tool плюс text editor tool
плюс агентный цикл дают связку «читать код → менять → гонять тесты → чинить».

### 🎯 Зачем тебе

Автономный дебаг с прогоном тестов, генерация отчётов из реальных данных на диске, DevOps-агент,
смотрящий логи и перезапускающий сервисы. Без Bash tool агент вынужден галлюцинировать результат
выполнения команды, с Bash tool — видит правду.

Разница между «моделью, рассуждающей о коде» и «моделью, реально его исполняющей» — не тонкость, а
принципиальная граница надёжности. Без исполняемого тула ответ вида «эта функция вернёт 42» —
предположение на основе чтения кода, которое может быть неверным из-за неучтённого edge case. С
Bash tool ответ становится проверяемым фактом: команда реально запущена, вывод реально получен, и
следующий шаг рассуждения строится на этом факте, а не на догадке о нём.

### 💻 Минимальный пример

```json
{
  "tools": [{ "type": "bash_20250124", "name": "bash" }],
  "messages": [{ "role": "user", "content": "Запусти тесты и почини первый упавший" }]
}
```

Модель присылает `tool_use` с `{"command": "pytest -x"}`, твой код исполняет это в **изолированной**
среде и возвращает `tool_result` со stdout — цикл повторяется, пока модель не сочтёт задачу решённой.

### ⚠️ Грабли

- **Без песочницы это RCE как фича** — модель может исполнить `rm -rf`, скачать что-то из интернета,
  прочитать секреты из окружения. Обязательна изоляция: контейнер без продовых credentials.
- **Долгие команды блокируют цикл** — тул должен уметь стримить вывод или таймаутить.
- **Нет автоматического подтверждения опасных команд на уровне API** — это логика твоей интеграции
  или Claude Code (permission prompts, allow-list).
