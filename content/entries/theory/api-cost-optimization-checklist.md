### ❓ Что это

Сводный чек-лист из пяти рычагов снижения счёта: **(1)** Prompt caching для стабильных больших
префиксов, **(2)** Batch API для несрочной фоновой обработки, **(3)** роутинг моделей (Haiku для
простых подзадач, Sonnet/Opus только там, где нужна глубина), **(4)** управление контекстом
(compaction/context editing), **(5)** service tier под тип трафика.

### 🎯 Зачем тебе

Продукт с реальным объёмом трафика, где счёт за API — заметная статья расходов. Типичная ошибка:
один и тот же большой Opus-запрос гоняется на все подзадачи, включая тривиальные, без кэширования и
батчинга.

### 💻 Минимальный пример

Порядок внедрения по убыванию окупаемости для большинства профилей трафика:

1. Prompt caching на системный промпт/кодовую базу — почти всегда безрисковый быстрый выигрыш.
2. Роутинг: маленькая модель на классификацию, большая — только на финальный синтез.
3. Batch для всего, что не требует ответа прямо сейчас.
4. Compaction/context editing для длинных агентных сессий.
5. Мониторинг `usage` в каждом ответе — без метрик оптимизация вслепую.

### ⚠️ Грабли

- **Оптимизация без замера вредна** — смена модели без A/B на качество может сэкономить на токенах и
  потерять на конверсии.
- **Кэширование и батчинг не складываются автоматически** — разные измерения экономии.
- **Частая ошибка** — не мониторить `cache_read_input_tokens`, непонятно, срабатывает ли кэш в проде.
