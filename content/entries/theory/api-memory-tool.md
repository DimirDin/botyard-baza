### ❓ Что это

Memory tool — клиентский инструмент (реализация на твоей стороне), дающий Claude читать и писать
файлы в директорию памяти между сессиями API. В отличие от контекстного окна, обнуляющегося с новым
разговором, файлы памяти переживают конец сессии — агент читает то, что сам записал раньше.

Формально это обычный `str_replace`-подобный тул с операциями `view`, `create`, `str_replace`,
`insert`, `delete` над файлами в песочнице памяти — «магия» персистентности лежит на твоей
инфраструктуре хранения.

### 🎯 Зачем тебе

Долгоживущий агент, который должен помнить факты о проекте/пользователе между запусками без
пересказа всей истории заново. Это тот же принцип, что и авто-память в этом Claude Code — файлы
в `~/.claude/.../memory/`.

### 💻 Минимальный пример

```json
{
  "tools": [{ "type": "memory_20250818", "name": "memory" }],
  "messages": [{ "role": "user", "content": "Продолжи работу над рефакторингом" }]
}
```

Модель вызывает `view` по каталогу памяти, находит файл вроде `progress/refactor-notes.md`, читает
и продолжает с того места, где остановилась, без ручного пересказа.

### ⚠️ Грабли

- **Ты отвечаешь за хранилище** — Anthropic не хранит эти файлы за тебя, бэкенд реализуешь сам.
- **Нет автоматической очистки устаревших заметок** — без гигиены памяти файлы разрастаются.
- **Модель может писать неверные факты в память** и потом доверять им — это состояние, а не истина,
  ревьюй периодически.
- **Многопользовательские системы требуют строгой изоляции директорий памяти по пользователю/проекту**
  — реализация тула сама по себе не гарантирует эту границу, если бэкенд перепутает пути, память
  одного пользователя может утечь в контекст другого.
