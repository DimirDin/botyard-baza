### ❓ Что это

Parallel tool use — способность модели вернуть несколько блоков `tool_use` в одном ответе вместо
строго одного за ход. Если задача естественно распадается на независимые подзапросы — «проверь
погоду в трёх городах», «посмотри статус пяти сервисов» — модель формулирует все вызовы сразу, ты
исполняешь их и возвращаешь результаты одним пакетом `tool_result` блоков.

### 🎯 Зачем тебе

Экономия на количестве round-trip'ов между тобой и API — вместо пяти последовательных обменов
получаешь один обмен с пятью тулами внутри. Особенно ощутимо на задачах с высокой латентностью
одного тула (внешний API, БД-запрос).

### 💻 Минимальный пример

```json
[
  { "type": "tool_use", "id": "1", "name": "get_weather", "input": { "city": "Москва" } },
  { "type": "tool_use", "id": "2", "name": "get_weather", "input": { "city": "Берлин" } }
]
```

Твой код исполняет оба (например, через `asyncio.gather`), и возвращает оба `tool_result` в
следующем сообщении, каждый со своим `tool_use_id`.

### ⚠️ Грабли

- **Не все версии одинаково охотно параллелят** — иногда нужен явный намёк в системном промпте.
- **Порядок результатов не гарантирован по времени** — сопоставляй строго по `tool_use_id`.
- **Ошибка одного параллельного тула не должна блокировать возврат остальных** — верни `tool_result`
  с пометкой ошибки именно для этого вызова.
- **Параллельные вызовы предполагают независимость друг от друга** — если один тул логически зависит
  от результата другого («сначала узнай ID пользователя, потом запроси его заказы»), это
  последовательная, а не параллельная задача; параллелить стоит только действительно независимые
  подзапросы, иначе легко получить рассинхрон между вызовами, использующими устаревшие данные друг
  друга.
