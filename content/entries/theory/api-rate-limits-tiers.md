### ❓ Что это

Rate limits в Anthropic Console — две оси: RPM (requests per minute) и TPM (tokens per minute),
лимиты растут по мере повышения тира аккаунта (автоматически, по накопленной истории трат, не по
запросу вручную). Превышение возвращает `429` с заголовками, сколько капасити осталось и когда
восстановится.

### 🎯 Зачем тебе

Продакшен-интеграция должна с первого дня закладывать обработку 429 — не экзотика, а нормальная часть
роста трафика. TPM может сработать раньше RPM при малом числе очень длинных запросов.

### 💻 Минимальный пример

```python
import time
def call_with_retry(fn, max_retries=5):
    for attempt in range(max_retries):
        try:
            return fn()
        except RateLimitError:
            time.sleep(2 ** attempt)
    raise RuntimeError("rate limit: retries exhausted")
```

Заголовки `anthropic-ratelimit-*-remaining` можно использовать проактивно — притормаживать очередь
до того, как реально прилетит 429.

### ⚠️ Грабли

- **Тир не повышается мгновенно после оплаты** — есть период накопления истории использования.
- **RPM и TPM считаются независимо** — низкая частота не спасает при очень длинных запросах.
- **Ретраи без backoff усугубляют проблему** — экспоненциальный backoff обязателен.
- **Проактивный throttling надёжнее реактивного ретрая** — если система заранее следит за оставшейся
  капасити по заголовкам ответа и сама притормаживает очередь исходящих запросов, это стабильнее, чем
  ловить `429` по факту и уже постфактум пытаться восстановиться — особенно на резких скачках трафика,
  где реактивный подход почти гарантированно ловит серию отказов перед тем, как выровняться.
