### ❓ Что это

Решает проблему масштаба: если у агента подключены сотни инструментов (десятки MCP-серверов),
передавать все их определения в системный промпт каждого запроса раздувает контекст — счёт растёт,
а релевантных конкретному вопросу от силы 2-3. Вместо статичного списка Claude сначала **ищет**
подходящий инструмент по текстовому запросу, и только затем получает полное определение и вызывает.

### 🎯 Зачем тебе

Настройка с большим числом MCP-серверов или корпоративным каталогом внутренних API-инструментов —
там, где заранее неизвестно, какие 3-5 тулов из 200 понадобятся для конкретного запроса.

### 💻 Минимальный пример

```json
{
  "tools": [{ "type": "tool_search_20251007", "name": "tool_search" }],
  "messages": [{ "role": "user", "content": "Найди последний коммит в репозитории botyard-baza" }]
}
```

Модель сначала вызывает `tool_search` с запросом вроде `"git commit history"`, получает список
подходящих тулов, затем вызывает найденный с конкретными аргументами.

### ⚠️ Грабли

- **Требует хороших описаний тулов** — поиск работает по текстовому сходству, плохо названный тул не
  найдётся (см. Writing tools for agents).
- **Дополнительный round-trip** — для 3-5 постоянно используемых тулов дешевле держать их в контексте
  напрямую.
- **Не заменяет продуманную группировку тулов** — плохая категоризация деградирует поиск до
  угадывания по ключевым словам.
