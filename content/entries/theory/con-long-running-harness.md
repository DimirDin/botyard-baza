### ❓ Что это

Harness — обвязка, позволяющая агенту работать часами или днями, переживая множество контекстных
окон подряд. Архитектура из двух ролей: **инициализатор** (создаёт план, начальное состояние) и
**кодинг-агент** (работает итеративно, оставляя артефакты — файлы прогресса, заметки, план —
которые следующая сессия подхватывает, не начиная с нуля).

#### Почему не просто «увеличить окно»

Даже с большим окном модель со временем накапливает шум (context rot). Harness решает это не
увеличением окна, а дисциплиной регулярного сброса состояния в персистентные артефакты и стартом
новой чистой сессии с ними.

### 🎯 Зачем тебе

Long-running задачи: многодневный рефакторинг, автономный research-процесс, проверяющий статус раз в
час. Без harness такие задачи либо упираются в лимит окна, либо требуют полного ручного пересказа
контекста при каждом перезапуске.

### 💻 Минимальный пример

```
progress/
  plan.md       # начальный план, редко меняется
  status.md     # что сделано, что осталось, актуализируется каждую сессию
  decisions.md  # принятые решения и почему
```

Новая сессия читает `status.md` и `decisions.md` вместо полной истории — тот же принцип, что у
Memory tool и compaction.

### ⚠️ Грабли

- **Артефакты должны обновляться активно** — без явной инструкции агент забывает записывать прогресс.
- **Слишком подробные заметки прогресса** сами становятся проблемой контекста.
- **Не решает дрейф решений** — без сверки с `decisions.md` можно получить несогласованный код между
  сессиями.
