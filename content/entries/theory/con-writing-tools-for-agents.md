### ❓ Что это

Практическое руководство по проектированию инструментов именно под LLM-агента, а не под обычный API-
контракт. Тул для агента читается моделью как текст (имя, описание, схема) прежде чем быть вызванным
— нечёткое описание приводит к тому, что агент либо не вызовет нужный тул, либо вызовет его с
неверными аргументами, даже если сам тул исправен.

#### Рекомендации из статьи

Называть тулы по действию, не по внутренней архитектуре (`search_customer_orders`, не `db_query_v2`).
Писать `description` как объяснение новому сотруднику, включая когда тул **не** стоит использовать.
Возвращать компактные, но информативные результаты. Консолидировать похожие операции в один тул с
параметром вместо десятка почти одинаковых.

### 🎯 Зачем тебе

Прямое применение при проектировании MCP-сервера или набора tool use функций. Плохо спроектированные
тулы — частая причина, почему агент «тупит», выбирая не тот инструмент.

### 💻 Минимальный пример

Плохо: `{ "name": "q", "description": "query db" }`

Хорошо:
```json
{
  "name": "search_customer_orders",
  "description": "Ищет заказы клиента по email или телефону. До 10 последних. Не используй для поиска по номеру заказа — для этого get_order_by_id.",
  "input_schema": { "type": "object", "properties": { "contact": { "type": "string" } }, "required": ["contact"] }
}
```

### ⚠️ Грабли

- **Слишком много похожих тулов** сбивает модель — сливай в один с параметром-переключателем.
- **Возврат сырых огромных ответов** тратит контекст — фильтруй под задачу заранее.
- **Отсутствие «когда не использовать» в описании** — модель выбирает тул не по назначению.
