### ❓ Что это
**Explore → Plan → Code → Commit** — рекомендованный Anthropic порядок работы с Claude Code на задачах сложнее пары строк: сначала агент исследует кодовую базу (Explore), затем предлагает план изменений (Plan), только потом пишет код (Code), и в конце фиксирует результат коммитом (Commit). Это не единственный способ работать с инструментом, но самый предсказуемый на нетривиальных задачах.

Отдельно от порядка шагов стоит **управление контекстом** — сессия Claude Code накапливает историю (прочитанные файлы, выполненные команды), и по мере роста задачи эта история может раздуваться, занимая всё больше «места» и постепенно снижая фокус на актуальной части работы.

### 🎯 Зачем тебе
Пропуск фазы Plan — самая частая причина, когда агент делает не то, что имелось в виду: без явного плана он может начать писать код по своей интерпретации задачи, а расхождение с твоими ожиданиями обнаружится только на этапе готового результата, когда переделка обходится дороже. Явный план перед кодом — это возможность поймать недопонимание на дешёвом этапе, а не после того, как правки уже внесены.

Управление контекстом напрямую влияет на качество ответов в длинных сессиях: чем длиннее история, тем больше в ней шума из уже неактуальных деталей, что может размывать фокус модели на текущей подзадаче — та же механика, что описана в статье про контекстное окно.

### 💻 Как это выглядит на практике
На практике цикл выглядит так:
1. **Explore** — просишь Claude Code изучить релевантную часть кода: «посмотри, как сейчас устроена авторизация» — без изменений, только чтение и понимание.
2. **Plan** — просишь явно сформулировать план: «предложи план добавления двухфакторной аутентификации» — получаешь список шагов до единой строчки кода.
3. **Code** — только после согласия с планом: «реализуй по этому плану» — теперь агент пишет код, уже опираясь на согласованную структуру.
4. **Commit** — просишь зафиксировать результат: «закоммить с понятным сообщением» — Claude Code сам формулирует commit message по факту изменений.

Для управления контекстом — команда `/clear` начинает сессию заново (полезно между несвязанными задачами, чтобы старая история не мешала новой), а обсуждение сложной подзадачи стоит явно резюмировать текстом перед тем, как двигаться дальше — это дешевле, чем тащить за собой полную историю исследования.

### ⚠️ Частая ошибка новичка
Просить Claude Code сразу писать код на сложную задачу, минуя явную фазу плана — экономия одного сообщения на входе часто оборачивается более долгой перепиской на исправления результата, который пошёл не в ту сторону.

Вторая ошибка — тащить одну и ту же сессию через несколько совершенно не связанных задач подряд без `/clear`. Накопленный контекст от предыдущей задачи может неявно влиять на решения по новой, даже если ты явно не ссылаешься на старую.
