### ❓ Что это
Интеграция Claude Code с локальным Docker через MCP-сервер даёт агенту типизированные инструменты для сборки образов, просмотра статуса контейнеров, чтения логов и перезапуска — не выходя из текущей сессии. Это конкретный практический пример того, как MCP (с уровня 3) применяется к реальной инфраструктурной задаче, а не просто к работе с текстом или кодом.

Вместо того чтобы объяснять агенту голосом, что происходит в контейнерах, он получает прямой доступ к тем же данным, что видишь ты сам в терминале — статусы, логи, метрики использования ресурсов.

### 🎯 Зачем тебе
Не нужно постоянно переключаться между терминалом с сессией Claude Code и соседней вкладкой с `docker compose logs -f`. Можно попросить агента самому найти причину падения контейнера по логам и поправить конфиг — весь цикл диагностики и исправления происходит внутри одного диалога, без ручного копирования логов туда-сюда.

Это особенно полезно при отладке сложных многоконтейнерных систем, где проблема может быть не в конкретном сервисе, а в их взаимодействии — агент может одновременно смотреть логи нескольких контейнеров и сопоставлять их по времени, что вручную делать утомительно.

### 💻 Как это выглядит на практике
Подключение через официальную команду:
```bash
claude mcp add docker -- npx -y <docker-mcp-пакет>
claude mcp list
```
Для командной работы — зафиксируй в `.mcp.json` в корне проекта (коммитится в git, применяется у всей команды):
```json
{
  "mcpServers": {
    "docker": { "type": "stdio", "command": "npx", "args": ["-y", "<docker-mcp-пакет>"] }
  }
}
```

![](/guide/4/guide-mcp-docker-detail.svg)

Практические сценарии, где это реально экономит время:
- **«Почему упал контейнер backend?»** — агент сам смотрит логи и exit-код, находит причину, предлагает фикс конфигурации.
- **«Собери образ и запусти локально»** — весь цикл сборки и запуска происходит по одной фразе, без ручного набора команд.
- **«Сравни, что изменилось в логах после последнего деплоя»** — агент может сопоставить состояние до и после изменений.

Как и с любым MCP-сервером, точное имя npm-пакета стоит сверять в актуальном каталоге MCP-серверов перед использованием — экосистема быстро развивается, конкретные пакеты появляются и устаревают.

### ⚠️ Частая ошибка новичка
Оставлять агента в headless-режиме с доступом к Docker без присмотра. Проброс доступа к Docker-сокету фактически даёт root-права на хосте — деструктивная команда очистки может снести все локальные базы и кэши образов разом, если агент интерпретирует задачу слишком буквально или ошибочно.

Вторая ошибка — давать этот доступ в проде без крайней необходимости. MCP-интеграция с Docker создана прежде всего для локальной разработки и отладки — для продакшен-инфраструктуры разумнее использовать более узкие, специально ограниченные права, а не тот же widescope доступ, что удобен на локальной машине.
